3 1. Qué es diferencias en diferencias

La idea básica de diferencias en diferencias es comparar cómo cambia un outcome en un grupo tratado antes y después de una intervención, y restarle cómo cambia el outcome en un grupo comparable no tratado.

En el caso clásico con dos grupos y dos periodos:

\[ ATT = \left(E[Y_{post}\mid D=1]-E[Y_{pre}\mid D=1]\right) - \left(E[Y_{post}\mid D=0]-E[Y_{pre}\mid D=0]\right) \]

El supuesto clave es la hipótesis de tendencias paralelas:

\[ E[Y_{post}(0)-Y_{pre}(0)\mid D=1] = E[Y_{post}(0)-Y_{pre}(0)\mid D=0] \]

Es decir, en ausencia del tratamiento, el grupo tratado habría evolucionado como el grupo de control.

3.1 1.1 Simulación de un DiD 2x2

n <- 1000

df_2x2 <- tibble(
  id = rep(1:n, each = 2),
  post = rep(c(0, 1), times = n),
  treated = rep(rbinom(n, 1, 0.5), each = 2),
  x1 = rep(rnorm(n), each = 2),
  x2 = rep(rbinom(n, 1, 0.4), each = 2)
) %>%
  mutate(
    D = treated * post,
    alpha_i = rep(rnorm(n), each = 2),
    time_effect = 0.5 * post,
    treatment_effect = 2,
    y = alpha_i + time_effect + 0.8 * x1 + 0.5 * x2 + treatment_effect * D + rnorm(n)
  )

4 2. Estructura de datos necesaria

Para estimar DiD necesitamos, como mínimo:

  • una variable de unidad (id), si tenemos panel;
  • una variable temporal (year, month, etc.);
  • un outcome (y);
  • una variable de tratamiento (D);
  • una variable de grupo de tratamiento o cohorte (G) si hay adopción escalonada;
  • covariables pretratamiento si queremos tendencias paralelas condicionales.

4.1 2.1 Simulación de un panel con adopción escalonada

n_id <- 1200
years <- 2010:2020

unit_data <- tibble(
  id = 1:n_id,
  gender = sample(c("Mujer", "Hombre"), n_id, replace = TRUE),
  sector = sample(c("Industria", "Servicios", "Construccion"), n_id, replace = TRUE,
                  prob = c(0.3, 0.5, 0.2)),
  size0 = rlnorm(n_id, meanlog = 2, sdlog = 0.5),
  x1 = rnorm(n_id),
  x2 = rbinom(n_id, 1, 0.4),
  G = sample(c(0, 2015, 2016, 2017, 2018), n_id, replace = TRUE,
             prob = c(0.35, 0.15, 0.18, 0.17, 0.15))
)

df <- crossing(id = 1:n_id, year = years) %>%
  left_join(unit_data, by = "id") %>%
  mutate(
    treated_ever = as.integer(G > 0),
    D = as.integer(G > 0 & year >= G),
    rel_time = if_else(G > 0, year - G, NA_integer_),
    unit_fe = rep(rnorm(n_id), each = length(years)),
    year_fe = 0.15 * (year - min(year)),
    # Tendencias heterogeneas por sector y genero
    trend_sector = case_when(
      sector == "Industria" ~ 0.05 * (year - min(year)),
      sector == "Servicios" ~ 0.02 * (year - min(year)),
      TRUE ~ 0.00 * (year - min(year))
    ),
    trend_gender = if_else(gender == "Mujer", 0.03 * (year - min(year)), 0),
    dynamic_te = case_when(
      D == 1 & rel_time == 0 ~ 0.8,
      D == 1 & rel_time == 1 ~ 1.2,
      D == 1 & rel_time >= 2 ~ 1.8,
      TRUE ~ 0
    ),
    y = unit_fe + year_fe + trend_sector + trend_gender +
      0.4 * x1 + 0.3 * x2 + 0.05 * log(size0) + dynamic_te + rnorm(n(), sd = 1)
  )

head(df)

Convención habitual en did::att_gt():

  • G = 0 indica unidades nunca tratadas.
  • G = año de tratamiento indica la primera fecha de tratamiento.

5 3. TWFE sin covariables

El modelo TWFE básico es:

\[ Y_{it}=\alpha_i + \lambda_t + \beta D_{it}+u_{it} \]

donde:

  • alpha_i son efectos fijos de unidad;
  • lambda_t son efectos fijos de tiempo;
  • D_it indica tratamiento.

8 6. Diseño escalonado: por qué cambia todo

En un diseño escalonado, distintas unidades reciben el tratamiento en distintos momentos. El parámetro natural ya no es un único ATT, sino:

\[ ATT(g,t) \]

donde:

  • g es la cohorte de tratamiento;
  • t es el periodo calendario.

El problema de TWFE en diseños escalonados es que puede comparar unidades tratadas tempranamente contra unidades tratadas más tarde, o incluso usar unidades ya tratadas como controles. Si los efectos son heterogéneos por cohorte o por tiempo desde tratamiento, el coeficiente TWFE puede mezclar efectos de forma difícil de interpretar.

10 8. Covariables en DiD

Meter covariables no es un acto mecánico. Cambia la hipótesis de identificación. Sin covariables, se pide tendencias paralelas incondicionales:

\[ E[\Delta Y(0)\mid D=1] = E[\Delta Y(0)\mid D=0] \]

Con covariables, se pide tendencias paralelas condicionales:

\[ E[\Delta Y(0)\mid D=1,X] = E[\Delta Y(0)\mid D=0,X] \]

10.1 8.1 Qué covariables usar

Buenas candidatas:

  • covariables pretratamiento;
  • niveles baseline de variables dinámicas;
  • outcomes pretratamiento;
  • características que afectan selección al tratamiento y evolución del outcome.

Evitar:

  • covariables post-tratamiento;
  • variables afectadas por el tratamiento;
  • kitchen sink controls sin justificación;
  • variables que destruyen overlap.

10.2 8.2 Estandarización

Estandarizar no cambia la identificación. Puede ayudar para interpretación, estabilidad numérica o modelos flexibles.

IPW no modela directamente el outcome. Estima la probabilidad de tratamiento condicional en covariables y pondera los controles para que se parezcan a los tratados.

13 11. RA: Regression Adjustment

RA estima el contrafactual de los tratados usando un modelo de outcome ajustado en los controles.

La idea es estimar:

\[ m_0(X)=E[\Delta Y\mid D=0,X] \]

Luego se predice qué habría ocurrido a los tratados en ausencia de tratamiento.

13.6 11.6 Qué comprobar en RA

  • Especificación del modelo de outcome.
  • Si la relación entre outcome y covariables es lineal o no.
  • Posible extrapolación para tratados fuera del soporte de controles.
  • Robustez a añadir interacciones o transformaciones de covariables.

14 12. DR: Doubly Robust DiD

DR combina IPW y RA. Estima tanto:

  • el propensity score p(X);
  • el modelo de outcome m0(X).

Tiene una propiedad importante: puede ser consistente si uno de los dos modelos auxiliares está bien especificado.

19 17. Buenas prácticas y errores comunes

19.1 17.1 Buenas prácticas

  1. Definir claramente el tratamiento y la primera fecha de tratamiento.
  2. Distinguir diseño 2x2, multiperiodo simple y adopción escalonada.
  3. Usar estimadores diseñados para adopción escalonada cuando corresponda.
  4. Graficar tendencias antes de estimar.
  5. No interpretar no-rechazo de pretrends como prueba definitiva.
  6. Usar covariables pretratamiento si la identificación condicional es más creíble.
  7. Revisar overlap si se usa IPW o DR.
  8. Reportar especificaciones alternativas.
  9. Analizar heterogeneidad solo si hay soporte suficiente dentro de subgrupos.
  10. Clusterizar al nivel adecuado.

19.2 17.2 Errores comunes

  1. Usar TWFE en diseños escalonados sin discutir heterogeneidad.
  2. Meter covariables post-tratamiento.
  3. Meter covariables invariantes en TWFE esperando identificar su coeficiente.
  4. Usar IPW sin mirar pesos u overlap.
  5. Usar RA con extrapolación fuerte.
  6. Pensar que DR arregla una mala elección de covariables.
  7. Hacer heterogeneidad con submuestras demasiado pequeñas.
  8. Exportar solo coeficientes agregados y no mostrar dinámica.

20 18. Workflow recomendado

Un flujo de trabajo razonable sería:

  1. Preparar datos: id, time, y, D, G, covariables baseline.
  2. Identificar si hay adopción escalonada.
  3. Graficar tendencias.
  4. Estimar TWFE como benchmark.
  5. Estimar event study moderno.
  6. Decidir si la hipótesis creíble es incondicional o condicional en covariables.
  7. Si se usan covariables, estimar DR como especificación principal.
  8. Comparar con IPW y RA.
  9. Evaluar overlap, balance y sensibilidad.
  10. Analizar heterogeneidad.
  11. Reportar tablas y gráficos.

22 20. Conclusión

La diferencia entre métodos no es solo computacional. Cada estimador construye el contrafactual de forma distinta:

  • TWFE compara cambios residuales dentro de una estructura lineal con efectos fijos.
  • IPW repondera controles para parecerse a tratados.
  • RA imputa el contrafactual usando modelos de outcome.
  • DR combina reponderación e imputación.

En diseños escalonados, el objeto natural es ATT(g,t). Por eso conviene usar estimadores diseñados para ese contexto, como did::att_gt() o event studies tipo sunab(), en vez de confiar únicamente en TWFE.

# 1. Qué es diferencias en diferencias

La idea básica de diferencias en diferencias es comparar cómo cambia un outcome en un grupo tratado antes y después de una intervención, y restarle cómo cambia el outcome en un grupo comparable no tratado.

En el caso clásico con dos grupos y dos periodos:

$$
ATT =
\left(E[Y_{post}\mid D=1]-E[Y_{pre}\mid D=1]\right)
-
\left(E[Y_{post}\mid D=0]-E[Y_{pre}\mid D=0]\right)
$$

El supuesto clave es la hipótesis de **tendencias paralelas**:

$$
E[Y_{post}(0)-Y_{pre}(0)\mid D=1]
=
E[Y_{post}(0)-Y_{pre}(0)\mid D=0]
$$

Es decir, en ausencia del tratamiento, el grupo tratado habría evolucionado como el grupo de control.

# 2. Estructura de datos necesaria

Para estimar DiD necesitamos, como mínimo:

- una variable de unidad (`id`), si tenemos panel;
- una variable temporal (`year`, `month`, etc.);
- un outcome (`y`);
- una variable de tratamiento (`D`);
- una variable de grupo de tratamiento o cohorte (`G`) si hay adopción escalonada;
- covariables pretratamiento si queremos tendencias paralelas condicionales.

## 2.1 Simulación de un panel con adopción escalonada

```{r simulate-staggered}
n_id <- 1200
years <- 2010:2020

unit_data <- tibble(
  id = 1:n_id,
  gender = sample(c("Mujer", "Hombre"), n_id, replace = TRUE),
  sector = sample(c("Industria", "Servicios", "Construccion"), n_id, replace = TRUE,
                  prob = c(0.3, 0.5, 0.2)),
  size0 = rlnorm(n_id, meanlog = 2, sdlog = 0.5),
  x1 = rnorm(n_id),
  x2 = rbinom(n_id, 1, 0.4),
  G = sample(c(0, 2015, 2016, 2017, 2018), n_id, replace = TRUE,
             prob = c(0.35, 0.15, 0.18, 0.17, 0.15))
)

df <- crossing(id = 1:n_id, year = years) %>%
  left_join(unit_data, by = "id") %>%
  mutate(
    treated_ever = as.integer(G > 0),
    D = as.integer(G > 0 & year >= G),
    rel_time = if_else(G > 0, year - G, NA_integer_),
    unit_fe = rep(rnorm(n_id), each = length(years)),
    year_fe = 0.15 * (year - min(year)),
    # Tendencias heterogeneas por sector y genero
    trend_sector = case_when(
      sector == "Industria" ~ 0.05 * (year - min(year)),
      sector == "Servicios" ~ 0.02 * (year - min(year)),
      TRUE ~ 0.00 * (year - min(year))
    ),
    trend_gender = if_else(gender == "Mujer", 0.03 * (year - min(year)), 0),
    dynamic_te = case_when(
      D == 1 & rel_time == 0 ~ 0.8,
      D == 1 & rel_time == 1 ~ 1.2,
      D == 1 & rel_time >= 2 ~ 1.8,
      TRUE ~ 0
    ),
    y = unit_fe + year_fe + trend_sector + trend_gender +
      0.4 * x1 + 0.3 * x2 + 0.05 * log(size0) + dynamic_te + rnorm(n(), sd = 1)
  )

head(df)
```

Convención habitual en `did::att_gt()`:

- `G = 0` indica unidades nunca tratadas.
- `G = año de tratamiento` indica la primera fecha de tratamiento.

# 3. TWFE sin covariables

El modelo TWFE básico es:

$$
Y_{it}=\alpha_i + \lambda_t + \beta D_{it}+u_{it}
$$

donde:

- `alpha_i` son efectos fijos de unidad;
- `lambda_t` son efectos fijos de tiempo;
- `D_it` indica tratamiento.

# 6. Diseño escalonado: por qué cambia todo

En un diseño escalonado, distintas unidades reciben el tratamiento en distintos momentos. El parámetro natural ya no es un único ATT, sino:

$$
ATT(g,t)
$$

donde:

- `g` es la cohorte de tratamiento;
- `t` es el periodo calendario.

El problema de TWFE en diseños escalonados es que puede comparar unidades tratadas tempranamente contra unidades tratadas más tarde, o incluso usar unidades ya tratadas como controles. Si los efectos son heterogéneos por cohorte o por tiempo desde tratamiento, el coeficiente TWFE puede mezclar efectos de forma difícil de interpretar.

Ejemplos:

- empresas que anticipan una subvención;
- trabajadores que anticipan un cambio regulatorio;

# 8. Covariables en DiD

Meter covariables no es un acto mecánico. Cambia la hipótesis de identificación. Sin covariables, se pide tendencias paralelas incondicionales:

$$
E[\Delta Y(0)\mid D=1] = E[\Delta Y(0)\mid D=0]
$$

Con covariables, se pide tendencias paralelas condicionales:

$$
E[\Delta Y(0)\mid D=1,X] = E[\Delta Y(0)\mid D=0,X]
$$

## 8.1 Qué covariables usar

Buenas candidatas:

- covariables pretratamiento;
- niveles baseline de variables dinámicas;
- outcomes pretratamiento;
- características que afectan selección al tratamiento y evolución del outcome.

Evitar:

- covariables post-tratamiento;
- variables afectadas por el tratamiento;
- kitchen sink controls sin justificación;
- variables que destruyen overlap.

## 8.2 Estandarización

Estandarizar no cambia la identificación. Puede ayudar para interpretación, estabilidad numérica o modelos flexibles.


# 10. IPW: Inverse Probability Weighting

IPW no modela directamente el outcome. Estima la probabilidad de tratamiento condicional en covariables y pondera los controles para que se parezcan a los tratados.

Para ATT, los controles con características más parecidas a los tratados reciben más peso.

## 10.1 IPW en DiD 2x2 con `DRDID`

```{r ipw-2x2}
ipw_2x2 <- ipwdid(
  yname = "y",
  tname = "post",
```

# 11. RA: Regression Adjustment

RA estima el contrafactual de los tratados usando un modelo de outcome ajustado en los controles.

La idea es estimar:

$$
m_0(X)=E[\Delta Y\mid D=0,X]
$$

Luego se predice qué habría ocurrido a los tratados en ausencia de tratamiento.

# 17. Buenas prácticas y errores comunes

## 17.1 Buenas prácticas

1. Definir claramente el tratamiento y la primera fecha de tratamiento.
2. Distinguir diseño 2x2, multiperiodo simple y adopción escalonada.
3. Usar estimadores diseñados para adopción escalonada cuando corresponda.
4. Graficar tendencias antes de estimar.
5. No interpretar no-rechazo de pretrends como prueba definitiva.
6. Usar covariables pretratamiento si la identificación condicional es más creíble.
7. Revisar overlap si se usa IPW o DR.
8. Reportar especificaciones alternativas.
9. Analizar heterogeneidad solo si hay soporte suficiente dentro de subgrupos.
10. Clusterizar al nivel adecuado.

## 17.2 Errores comunes

1. Usar TWFE en diseños escalonados sin discutir heterogeneidad.
2. Meter covariables post-tratamiento.
3. Meter covariables invariantes en TWFE esperando identificar su coeficiente.
4. Usar IPW sin mirar pesos u overlap.
5. Usar RA con extrapolación fuerte.
6. Pensar que DR arregla una mala elección de covariables.
7. Hacer heterogeneidad con submuestras demasiado pequeñas.
8. Exportar solo coeficientes agregados y no mostrar dinámica.

# 18. Workflow recomendado

Un flujo de trabajo razonable sería:

1. Preparar datos: `id`, `time`, `y`, `D`, `G`, covariables baseline.
2. Identificar si hay adopción escalonada.
3. Graficar tendencias.
4. Estimar TWFE como benchmark.
5. Estimar event study moderno.
6. Decidir si la hipótesis creíble es incondicional o condicional en covariables.
7. Si se usan covariables, estimar DR como especificación principal.
8. Comparar con IPW y RA.
9. Evaluar overlap, balance y sensibilidad.
10. Analizar heterogeneidad.
11. Reportar tablas y gráficos.

# 20. Conclusión

La diferencia entre métodos no es solo computacional. Cada estimador construye el contrafactual de forma distinta:

- TWFE compara cambios residuales dentro de una estructura lineal con efectos fijos.
- IPW repondera controles para parecerse a tratados.
- RA imputa el contrafactual usando modelos de outcome.
- DR combina reponderación e imputación.

En diseños escalonados, el objeto natural es `ATT(g,t)`. Por eso conviene usar estimadores diseñados para ese contexto, como `did::att_gt()` o event studies tipo `sunab()`, en vez de confiar únicamente en TWFE.
